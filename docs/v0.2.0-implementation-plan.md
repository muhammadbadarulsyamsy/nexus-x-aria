# v0.2.0 Implementation Plan Draft

## Purpose

This document plans the implementation sequence for NEXUS-X ARIA v0.2.0.

This is not runtime code.

## Implementation Theme

```text
Explainable Scoring
```

## Files Likely to Change During Runtime Patch

Minimum runtime patch:

```text
worker.js
package.json
docs/explainable-scoring-v0.2.0.md
docs/release-snapshot-v0.2.0.md
```

Likely follow-up docs:

```text
README.md
CHANGELOG.md
docs/publication-docs-index.md
postman/nexus-x-aria.postman_collection.json
```

## Suggested Implementation Sequence

### Step 1 — Version Update

Update:

```text
const VERSION = "0.2.0";
```

Update package version:

```text
0.2.0
```

### Step 2 — Add Explanation Containers

In scoring function, initialize:

```text
rule_hits = []
score_breakdown = []
false_positive_notes = []
```

### Step 3 — Add addSignal Helper

Concept:

```text
addSignal(rule_id, category, points, severity, reason, red_flag)
```

This helper should update:

```text
risk_score
rule_hits
score_breakdown
red_flags
```

Benefits:

```text
[✓] score and explanation stay aligned
[✓] fewer duplicate code paths
[✓] easier testing
```

### Step 4 — Convert Existing Rules

Convert existing scoring logic gradually:

```text
URL checks
domain keyword checks
wallet request checks
signing request checks
secret recovery checks
approval risk checks
token contract checks
social link checks
chain checks
Indonesian scam phrase checks
```

### Step 5 — Add input_quality

Assess missing fields:

```text
project_name
official_url
description
required_tasks
chain
token_contract
social_links
```

Suggested level logic:

```text
complete = most important fields present
partial = some important fields missing
weak = many important fields missing
```

### Step 6 — Add confidence_note

Always include:

```text
This result is based on heuristic text and metadata analysis only. It may contain false positives or false negatives.
```

### Step 7 — Add false_positive_notes

Add notes for safe contexts:

```text
educational approval warning
known chain-related sync context
```

### Step 8 — Update Test Pack

Increase test count:

```text
20 → 25
```

Add tests:

```text
TC21 score_breakdown exists
TC22 rule_hits include wallet/signing rules
TC23 input_quality detects missing token_contract
TC24 educational approval warning creates false_positive_notes
TC25 confidence_note exists
```

### Step 9 — Update OpenAPI

Update:

```text
info.version = 0.2.0
response schema fields
example response
```

### Step 10 — Update Try Page Compatibility

Ensure `/try` still displays JSON result.

No need for major UI redesign.

### Step 11 — Live Deployment Flow

After local patch and commit:

```text
copy worker.js to clipboard
paste into Cloudflare dashboard
deploy
verify live endpoints
```

Do not use Wrangler in Termux Android.

### Step 12 — Live Validation

Verify:

```text
GET /health
GET /openapi.json
GET /test-pack
GET /try
GET /examples
POST /score-airdrop
```

### Step 13 — GitHub Release

Only after live validation passes:

```text
gh release create v0.2.0
```

### Step 14 — Memory Database

After release:

```text
NEXUS_X_ARIA_MEMORY_DATABASE_v1.0
```

## Do Not Include in v0.2.0

```text
API key
login
database
external API
blockchain explorer integration
domain reputation feed
wallet simulation
RapidAPI
monetization
Postman public publish
```

## Risk Controls

Main risks:

```text
score_breakdown does not match score
rule_hits missing triggered rules
OpenAPI version mismatch
test pack failure
Try Page breakage
```

Controls:

```text
use addSignal helper
add test cases
verify live after deploy
do not release before validation
```
