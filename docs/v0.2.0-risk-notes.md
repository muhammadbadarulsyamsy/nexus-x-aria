# v0.2.0 Risk Notes

## Purpose

This document records risks and boundaries for the planned v0.2.0 Explainable Scoring update.

## Main Product Risk

Explainability can make ARIA look more authoritative than it is.

To avoid overclaiming, every response should include:

```text
confidence_note
```

Recommended note:

```text
This result is based on heuristic text and metadata analysis only. It may contain false positives or false negatives.
```

## Technical Risks

### Risk 1 — score_breakdown mismatch

Problem:

```text
risk_score changes but score_breakdown does not explain all points.
```

Mitigation:

```text
Use addSignal helper.
Every scoring rule should add points and explanation at the same time.
```

### Risk 2 — rule_hits incomplete

Problem:

```text
A rule affects score but does not appear in rule_hits.
```

Mitigation:

```text
Only add scoring through addSignal.
```

### Risk 3 — OpenAPI version mismatch

Problem:

```text
/health says 0.2.0 but /openapi.json says older version.
```

Mitigation:

```text
Use VERSION constant for OpenAPI info.version where possible.
Add verification step.
```

### Risk 4 — Test Pack instability

Problem:

```text
New tests fail due to response shape changes.
```

Mitigation:

```text
Add tests gradually and verify locally/live before release.
```

### Risk 5 — Response too verbose

Problem:

```text
score response becomes too long for simple users.
```

Mitigation:

```text
Keep reasons concise.
Do not add unnecessary nested data.
```

## Safety Risks

Do not add:

```text
wallet connection
private key input
seed phrase input
API token input
cookie input
external paid service
```

## Business Risks

Do not add:

```text
monetization
RapidAPI
paid API keys
billing
payout setup
subscription
```

## Recommended Boundary

v0.2.0 should improve only:

```text
explainability
testability
response transparency
developer readability
```

It should not change ARIA into a paid security product.
